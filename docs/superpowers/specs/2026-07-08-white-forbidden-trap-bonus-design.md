# 백의 금수 유도 사(四) 보너스 — 설계

## 배경

사용자가 "AI가 흑일 때 장목을 아예 생각하지 않고 두는" 문제를 제보했다. 원인 분석 결과, 근본 원인은 `getFourThreats(board, row, col, player)`가 `player`의 완성 지점을 셀 때 `player`가 흑이어도 그 완성 지점 자체가 흑에게 금수(33/44/장목)인지 확인하지 않는 것이었다 — 그래서 완성 지점 2곳 중 1곳이 실제로는 금수라 상대가 못 막는 게 아니라 안 둬도 되는 자리인데도 "더블사(확정승리)"로 오판했다. `getFourThreats`에 `player === 1`일 때 완성 지점을 `checkForbidden`으로 걸러내는 필터를 추가해 이 버그를 수정했다(별도 커밋).

이 버그 수정과 함께, 사용자가 "AI가 백일 때 상대(흑)의 금수를 유도하는 로직"을 추가로 요청했다. 확인해보니 `searchVCF`(`aiEngine.js` 약 350행)에는 이미 이 개념이 일부 구현돼 있다 — 강제수순(VCF) 탐색 도중 사(四)의 유일한 완성 지점이 흑에게 금수면 "상대가 못 막으니 확정승리"로 처리한다. 하지만 이 인식은 `searchVCF`가 성립하는 좁은 상황(사가 연속으로 이어지는 강제수순)에만 있고, 일반 탐색이 매 후보를 채점하는 `scoreCell`에는 전혀 반영되어 있지 않다 — 즉 아직 강제수순이 아닌 평범한 국면에서 "이 자리에 사를 만들면 흑이 합법적으로 못 막는다"는 정보를 검색이 전혀 모른다.

## 참고: 과거 비슷한 시도들의 교훈

`docs/TROUBLESHOOTING.md` #14·#15에 "상대의 다단계 강제수순을 미리 감지"하는 기능을 두 번(필터 방식, 힌트 방식) 시도했다가 자가대국 10판에서 4승6패로 밀려 되돌린 기록이 있다. 결론은 "기존 반복심화 탐색이 이미 스스로 찾아내는 경우가 많아 명시적 사전 감지의 실익이 적다"는 것이었다.

이번 기능은 그 시도들과 다르다: 상대의 다단계 수순을 미리 탐색하는 게 아니라, **이미 계산 중인 사(四) 완성 지점 하나**(`getFourThreats`가 어차피 계산하는 값)에 `checkForbidden` 확인 한 번만 추가하는 것이라 범위가 훨씬 좁고, 새 탐색 구조를 만들지 않는다. `scoreCell`은 후보 정렬용 힌트일 뿐 탐색의 정확성 자체를 바꾸지 않으므로 회귀 리스크도 낮다.

## 설계

**대상 함수:** `client/src/utils/aiEngine.js`의 `scoreCell(board, row, col, player)`

**변경 내용:** 기존 포크 보너스 블록(`myFourDirs >= 2` / `myFourDirs >= 1 && myThreeDirs >= 1`) 뒤에, 다음 조건을 추가한다.

```js
// 상대가 흑이고 이 자리가 어느 방향으로든 사(四)를 만드는 후보라면(myThreeDirs>=1 —
// analyzeDirection은 착수 "전" 기준으로 세므로 실제 사가 되는 자리도 3목+빈칸2로
// 잡힌다: 그 빈칸 중 하나가 바로 이 후보 자신이기 때문. myFourDirs는 이미 착수 없이도
// 4목+빈칸1인 경우(=즉시 5목 승리)라 이 용도엔 안 맞음), 실제로 착수해 만들어지는
// 사의 완성 지점을 getFourThreats로 계산하고, 유일한 완성 지점이 흑에게 금수(33/44/
// 장목)인지 확인한다. 금수라면 흑은 이 자리를 합법적으로 막을 수 없으므로 사실상
// 확정승리에 준한다 — searchVCF는 이미 강제수순 안에서 이 판단을 하지만(약 350행),
// 강제수순이 아직 성립하지 않은 평범한 국면에서는 이 정보가 후보 정렬에 전혀
// 반영되지 않았다.
if (opp === 1 && myFourDirs === 0 && myThreeDirs >= 1) {
  board[row][col] = player
  const completions = getFourThreats(board, row, col, player)
  board[row][col] = 0
  if (completions.length === 1 && checkForbidden(board, completions[0].row, completions[0].col) !== null) {
    score += 30000
  }
}
```

- `myFourDirs === 0 && myThreeDirs >= 1`로 한정하는 이유: 처음엔 `myFourDirs === 1`로 설계했으나 구현 중 실제로 검증해보니, `analyzeDirection`은 착수 "전" 보드 상태를 기준으로 세기 때문에 `mine.fourWindows`는 "이미 4목이 있고 빈칸이 이 후보 자신 하나뿐" — 즉 착수하면 곧장 5목 완성(즉시 승리)인 경우에만 성립한다. 반대로 "이 후보를 두면 사(四)가 만들어진다"(3목 + 빈칸 2개, 그중 하나가 이 후보 자신)는 `mine.threeWindows`로 잡힌다. `myFourDirs >= 2`(오픈사/더블사에 준하는 즉시 승리권)는 이미 +80000으로 최우선 처리되므로 중복 적용을 피하려 `myFourDirs === 0`도 같이 확인한다.
- `completions.length === 1`로 한정하는 이유: 완성 지점이 2곳 이상이면 오픈사(사실상 확정승리, 별도 처리 대상)이고, 이 블록은 "완성 지점이 하나뿐인 평범한 사인데 그 하나가 하필 흑에게 금수라 사실상 못 막는" 특수 케이스만 노린다.
- 비용: `getFourThreats`+`checkForbidden` 호출이 추가되지만 `myThreeDirs >= 1`일 때만(이 후보가 실제로 사를 만드는 드문 경우) 실행되므로 매 후보 채점마다 도는 비용이 아니다.
- `score += 30000`: 오픈사(+80000)보다는 낮고 일반 사(+10000)보다는 훨씬 높은 값 — "사실상 확정승리"라는 실질에 맞춰 오픈사에 준하되 완전히 같은 값은 피했다(오픈사는 진짜 무조건 승리, 이쪽은 흑이 굳이 그 자리에 두지 않고 다른 수로 게임을 포기하지 않는 이상 승리라 아주 근소하게 낮춤).

### 건드리지 않는 것

- `searchVCF`, `findCriticalDefenseCells`, `negamax`/`rootSearch`의 `extend` 판단 로직 — 전부 기존 그대로
- 삼(三) 단계의 금수 유도 판정 — 범위 밖(위 대화에서 사용자와 합의)

## 검증 계획

1. 격리 재현: 흑이 사(四)를 만들되 유일한 완성 지점이 장목/33/44인 국면을 손으로 구성해 `scoreCell`이 실제로 +30000을 더하는지 확인
2. P0 회귀: 기존 스팟체크(오픈쓰리 방어, 자기 VCF 우선, 오프닝북 조회)
3. `npx vite build`
4. 10판 자가대국(현재 HEAD vs 신규, 색 교대, 동일 조건) — 별도 커밋인 장목 버그 수정 위에 이 기능을 얹은 버전으로 비교

## 채택 기준

격리 재현이 기대대로 동작하고 P0/build 통과, 자가대국에서 뚜렷한 퇴보가 없으면 채택. 자가대국이 노이즈로 흔들리는 경우(과거 #14/#15처럼 고정 오프닝 소표본 특성) 격리 재현 성공과 빌드 통과를 우선 근거로 삼는다 — 이 기능은 새 탐색을 추가하는 게 아니라 기존에 이미 계산하던 값에 이미 존재하는 규칙(금수)을 한 번 더 확인하는 것뿐이라, 자가대국 승패보다 "이 특정 상황을 정확히 인식하는지"가 채택의 핵심 근거다.
