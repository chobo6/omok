## AI 엔진 강화 분석·수정 내역 (2026-07-03)

사용자 요청으로 "단일 난이도, 최대한 전문가 수준"을 목표로 AI를 강화하기로 하고, 참고 삼아 두 오목 엔진 리포를 리서치했습니다.

- **[Piskvork](https://github.com/plastovicka/Piskvork)**: 확인 결과 AI 엔진이 아니라 Gomocup 프로토콜 GUI/심판 툴이었음. 예제 봇은 랜덤 착수 수준. `source/pbrain/alfabeta.cpp`(~400줄)만 컴팩트한 알파베타+VCF 참고용으로 볼만했지만 Rapfi 대비 얻을 게 없어 실질적으로 기여한 내용 없음
- **[Rapfi](https://github.com/dhbloo/rapfi)**: Gomocup 상위권 엔진. 신경망(NNUE)·MCTS는 학습된 가중치와 ML 런타임이 필요해 브라우저 JS로 이식 불가 — 제외. 대신 알파베타 계열의 **구조적 기법**(반복심화, Transposition Table, 매 노드 후보 재정렬, 조합 패턴 평가, VCF 상대방어 탐색 등)은 대부분 순수 JS로 이식 가능해서 우선순위별로 정리 후 단계적으로 적용하기로 함

### 1단계 적용 완료 (`client/src/utils/aiEngine.js`)
- 최대화/최소화를 따로 두던 `minimax`를 표준 `negamax`(관점별 부호 반전 하나로 통일) 구조로 교체
- **반복심화(Iterative Deepening)**: 고정 depth-3 → 시간 예산(`TIME_BUDGET_MS=2000ms`) 안에서 depth 1→2→3…로 점점 깊게, 완료된 depth만 채택
- **Transposition Table**: Zobrist 해싱(칸×플레이어별 난수 XOR 증분 계산) + `Map` 기반 캐시(이번 탐색 호출 동안만 유지)
- **매 노드 후보 재정렬**: 기존엔 루트에서만 정렬하고 재귀 내부는 그대로 썼는데, 모든 노드에서 재정렬하도록 수정 (Rapfi 조사 중 발견한 기존 코드의 실제 비효율)

### 검증
- 실제 시나리오 재현 테스트: 회귀(P0 열린사/열린삼 방어), 적법성, 시간예산 준수(≤2500ms) 전부 통과
- **깊이 실측**: 초반 국면 depth 5, 중반 혼잡 국면 depth 7까지 도달 (기존 고정 depth-3 대비 2배 이상) — TT 12,000~22,000개 엔트리 생성돼 실제로 재사용되는 것 확인
- 평가함수 대칭성(`evaluate(board,1) === -evaluate(board,2)`) 검증 통과 — negamax 전환 시 흔한 부호 버그 없음 확인
- `npx vite build` 정상 통과

### 2단계 적용 완료 (같은 날, `client/src/utils/aiEngine.js`)
- **VCF-defend(`hasImmediateWin`)**: `searchVCF`가 사(四)를 강제할 때마다 상대에게 이미 즉시 승리 수가 없는지 확인. 검증 테스트를 구성하던 중 "오픈사 주장 분기가 이 확인을 우회하는" 순서 버그를 추가로 발견해 함께 수정 (`docs/TROUBLESHOOTING.md` #6)

### 2단계 검증

## AI 엔진 3단계 시도 (2026-07-03, 바운딩박스는 채택 / Killer·History·PVS는 되돌림)

3단계 항목(Killer Move/History Heuristic, PVS, 바운딩박스) 세 개를 한 번에 구현한 뒤 자가대국으로 검증하는 과정에서, 계획대로 되지 않은 두 가지를 발견해 최종적으로 바운딩박스만 채택했습니다. 자세한 원인 분석은 `docs/TROUBLESHOOTING.md` #8 참고.

- **바운딩박스 증분 관리**: `getCandidates`가 매 노드마다 보드 15×15 전체를 스캔하던 것을, 지금까지 놓인 돌의 바운딩박스(+여유 2칸)만 스캔하도록 축소. 착수마다 `expandBBox`로 갱신, 루트에서 1회만 전체 스캔. **자가대국(800ms 예산) 8판 8승**으로 실질적인 강화를 확인해 채택.
- **Killer Move / History Heuristic**: 처음 구현에서 킬러 보너스(+50000)가 `scoreCell`의 실제 사·삼 위협 점수를 역전시켜 가지치기가 오히려 나빠지는 걸 고정 depth 노드 수 비교로 발견 → `scoreCell*1000` 위에 999 미만으로만 얹어(수학적으로 전술 순위를 절대 뒤집지 못하는) 안전하게 재보정. 그런데도 자가대국에서 killer+history만 켰을 때 **1승 9패**로 크게 밀림 — `scoreCell` 자체가 이미 강한 전술 신호라 타이브레이킹 여지가 적은데, 매 노드 킬러/히스토리 테이블 조회 오버헤드는 항상 드니 순손실로 판단해 제거.
- **PVS(Principal Variation Search)**: 자가대국에서 **0승 8패**로 훨씬 크게 밀림. Transposition Table과 결합했을 때, 널윈도우 재탐색 직후 같은 해시로 즉시 재호출하는 "풀윈도우 재탐색"이 방금 저장된 좁은 윈도우 기준 TT 항목을 그대로 재사용해버려 사실상 재탐색이 무력화되는(원래 얻으려던 정확한 값을 못 구하는) 문제로 추정 — 제거.

### 검증
- P0 회귀(오픈사/오픈삼 방어, 즉시승리 우선, 사 생성 포착), 적법성, 시간예산, `vite build` 전부 정상

## Yixin 대국 벤치마크 도구 (2026-07-03)

지금까지 AI 강화 검증은 전부 "구버전 vs 신버전" 자가대국(미러매치)이었는데, 같은 엔진 계열끼리는 같은 맹점을 공유해 진짜 개선을 놓칠 수 있다는 문제가 있었다 (VCT 6승 6패, 평가함수 정교화 16승 14패 — 둘 다 논리적으로는 타당한데 미러매치에서 중립으로 나와 실제 효과를 판단 못 함). 로컬에 설치된 독립 엔진 **Yixin**(Kai Sun, Gomocup 상위권, `C:\Yixin`)을 고정 기준점으로 삼아 이 문제를 해결하기로 함.

- `tools/bench-vs-yixin.mjs` 신규 작성 — Yixin의 `engine.exe`(Gomocup pbrain 프로토콜, stdin/stdout)를 자식 프로세스로 띄워 `aiEngine.js`의 `getAIMove`와 실시간으로 맞대결시키는 헤드리스 Node 스크립트. GUI 불필요.
- **좌표 순서 실측 검증**: pbrain 좌표는 `X,Y`(열,행)로 `board[row][col]`과 순서가 다름 — 세로 열린삼을 인위적으로 만들어 Yixin이 정확히 세로 방향의 빈 칸을 막는지 확인해 매핑(`x=col, y=row`)이 맞음을 확인
- **버그 발견·수정**: 오프닝 다양화를 위해 보드에 몇 수를 미리 놓고 시작하려 했는데, `BEGIN`/`TURN` 프로토콜을 거치지 않고 보드에 직접 놓은 돌은 Yixin이 그 존재를 전혀 모름 → Yixin이 그 자리에 자기 수를 두면서 충돌해 즉시 "무승부"로 잘못 끝나는 버그 발생 (6판 중 4판에서 재현). 모든 착수를 예외 없이 `BEGIN`/`TURN`으로 실시간 전달하는 구조로 재작성해 해결 — 자가대국 스크립트와 달리 상대가 외부 프로세스라 상태를 마음대로 조작할 수 없다는 게 원인
- **베이스라인 측정(1차, 자유룰)**: 현재 HEAD(`edf491f`) 기준 6판(백 3판+흑 3판, 각 2000ms/수, 자유룰) 실행 → **Yixin 6승 0패**
- **알려진 한계**: 내 엔진이 흑(선공)일 때는 첫 수가 결정론적이라 여러 판이 동일 수순으로 겹칠 수 있음 (오프닝 랜덤화가 "상대 첫수에 대한 응수"에만 있고 "완전히 빈 보드에서의 내 첫수"에는 없음). 또한 반복심화가 `Date.now()` 벽시계 기준이라 실행할 때마다 미세하게 다른 depth에서 끊길 수 있어 완전히 결정론적이지는 않음
- `ENGINE_MODULE_PATH`를 커맨드라인 인자로 넘길 수 있게 해둬서(`node tools/bench-vs-yixin.mjs <경로>`) 스크래치패드의 실험용 엔진 사본과 곧바로 비교 가능

### 벤치마크 규칙 수정: 자유룰 → 렌주룰 (2026-07-03)

자유룰 벤치마크에서 "탐색 시간을 2000→2500ms로 늘렸더니 0승 6패 → 3승 3패"라는 고무적인 결과가 나왔지만, 자세히 보니 **이긴 판이 전부 내 엔진이 흑(선공)일 때뿐**이고 백(프로덕션 조건)은 여전히 0승이었다. 오목/렌주에서 흑 선공은 자유룰에서 원래 압도적으로 유리하다(렌주룰에 흑 전용 금수가 있는 이유 자체가 이 이점을 상쇄하기 위해서). 즉 그 "개선"은 AI가 세져서가 아니라 자유룰의 흑 이점을 흑 쪽 3판이 흡수한 것뿐이었을 가능성이 큼 — 프로덕션은 렌주룰이라 이 자유룰 벤치마크가 백 성능을 왜곡해서 보여주고 있었다.

- `INFO rule 4`(RULE_RENJU)로 Yixin을 렌주룰로 전환. 인위적으로 33(더블삼) 자리를 만들어 Yixin이 후보 1순위였던 그 자리를 실제로 피하는 것을 확인해 프로토콜이 제대로 먹히는 것을 검증
- 사용자 방침: **앞으로 AI가 흑/백을 선택해서 둘 수 있게 할 예정이라 두 색 모두 전문가 수준이어야 하고, 규칙은 항상 렌주룰로 통일한다.** 다만 지금은 AI가 백만 두므로(`aiEngine.js`에 흑 금수 회피 로직이 아직 없음) 벤치마크도 `MY_COLOR=2`(백) 고정으로만 돌리기로 함(흑 지원은 추후 과제)
- **렌주룰+백으로 재검증한 결과**: 2000ms/2500ms/4000ms **세 예산 모두 6전 전패, 수순 길이도 21~27수로 동일한 범위** — 자유룰에서 봤던 "2500ms 개선"은 완전히 사라짐. **즉 탐색 시간을 2배로 늘려도(2000→4000ms) 백은 전혀 달라지지 않는다**는 게 정직한 결론. 이전 세션에 중단됐던 "탐색 깊이/시간 자체가 실력에 영향을 주는가" 진단이 이걸로 일단락됨 — 이 예산 범위(2000~4000ms) 안에서는 아니라는 결과
- 시사점: 병목은 탐색 시간 자체가 아니라 **평가함수 품질**이나 **오프닝 단계 대응**, 혹은 Yixin 자체의 알고리즘적 우위(패턴 기반 평가, 오프닝 북 등) 쪽에 있을 가능성이 높음. 다음 방향을 정하려면 사용자와 논의 필요

### 패턴(열림/막힘) 평가함수 시도 및 되돌림 (2026-07-03)

Yixin의 실제 탐색 진단 메시지(`INFO`/`TURN` 응답의 MESSAGE 라인)를 분석하고 공개된 아키텍처 특징(반복심화+TT, VCF/VCT 전용모드, 멀티스레드, 오프닝 북)과 비교한 결과, 가장 유력한 차이는 **평가함수가 열린삼/막힌삼을 구분 못 한다는 점**으로 좁혀짐. `boardScore`를 "5칸 윈도우 내 돌 개수만 세는" 방식에서 `classifyLine`(인접 윈도우 매치를 그룹으로 묶어 열림/막힘 판정) 기반으로 교체 — 열린삼 점수를 막힌삼 대비 대폭 상향(5000 vs 150, 기존엔 사실상 거의 동급이었음).

### 반사적 방어 → 탐색 기반 방어로 전환 시도 및 되돌림 (2026-07-03)

Yixin 벤치마크에서 실제로 진 판 하나를 골라 백의 매 수 직전 `evaluate()` 점수를 추적한 결과, 4수째까지는 무난하다가(0→490) 5수째에서 갑자기 -702로 급락하는 지점을 발견. 그 지점의 `findCriticalDefenseCells` 호출을 직접 확인하니 critical cell이 2개 검출됐는데, `getAIMove`의 "위급 방어" 분기는 이럴 때 **`iterativeDeepeningSearch`(진짜 반복심화 탐색)를 아예 호출하지 않고** `scoreCell` 한 번만으로 즉시 반환하는 구조였음. 같은 판을 더 추적하니 백이 둔 10수 중 7수가 이 반사 분기(탐색 생략)였음 — VCT·시간예산·평가함수를 아무리 고쳐도 효과가 없었던 이유가 설명됨(그 코드들은 전부 `iterativeDeepeningSearch` 경로에 있는데 그 경로 자체가 거의 안 불렸음).

- 수정: `criticalCells`를 찾으면 즉시 반환하는 대신, `orderCandidates`에 `forcedCells` 파라미터를 추가해 이 자리들이 `MAX_CANDIDATES_PER_NODE`에 밀려 잘리지 않도록 최우선 순위로만 고정하고, 실제 선택은 `iterativeDeepeningSearch`가 몇 수 앞을 보고 결정하게 함(재귀 호출인 `negamax` 내부까지는 `forcedCells`를 전달하지 않음 — 루트 결정 로직만 바꾼 것)
- P0 회귀(열린사 방어, 자기 VCF 우선)·`vite build` 통과. 실전 재현: 문제의 국면에서 2~3ms만에 반사적으로 결정하던 게 실제로 2000ms 탐색을 쓰도록 바뀐 것 확인(이 특정 국면에선 결과 수는 우연히 동일)
- **자가대국 12판(300ms): 8승 4패** — 이 세션 처음으로 나온 뚜렷한 비대칭 결과(이전 세 시도는 전부 6승 6패)
- Yixin 벤치마크 1차(6판): 0승 6패지만 수순이 15/19/29/25/27/19로 편차가 커서(베이스라인은 21~23으로 고름) 표본 부족 의심 → **2차 6판 추가, 베이스라인도 6판 추가해 각 12판으로 재비교**
- **최종(12판 vs 12판)**: 베이스라인 0승 12패(평균 23.0수), 수정본 0승 12패(평균 23.17수) — **거의 동일**. 자가대국 8승4패라는 뚜렷한 신호가 있었음에도 Yixin 상대로는 표본을 늘리자 사라짐. `git restore`로 되돌림(미커밋)
- **교훈**: 근본 원인 진단이 아무리 탄탄해도(직접 국면 추적으로 확인), 자가대국이 아무리 뚜렷해도(8승4패, 이 세션 최초) Yixin 벤치마크가 표본이 작으면(6판) 우연히 낙관적으로 보일 수 있다 — 표본을 늘리기 전까진 결론 내리지 말 것

### 종합: 네 번의 연속 시도 전부 Yixin 상대 무효 (2026-07-03)

VCT, 시간예산 2배, 패턴 평가함수, 탐색기반 방어 — 네 가지 전부 이론적으로 타당하고 최소 하나 이상의 검증(자가대국 또는 정적 검증)을 통과했지만 Yixin 벤치마크에서 전부 베이스라인과 실질적으로 동일했다. 공통점: 패배가 항상 15~29수 안에서 끝남. 남은 가설: (1) Yixin과의 실력차가 이런 국지적 수정으로는 못 메울 만큼 근본적(네이티브 코드 속도, 오프닝 북, 수십 년 축적된 패턴 데이터베이스 등), (2) 오프닝~초반 단계의 수 선택 품질 자체(지금은 랜덤 응수뿐, 이론적 기반 전무)가 진짜 병목일 가능성.

### 오프닝 위치 평가(중앙 근접 가산점) 시도 및 되돌림 (2026-07-04)

사용자 방침: Yixin은 반드시 이겨야 할 상대가 아니라 유의미한 개선을 판별하기 위한 기준점. 오프닝 단계(돌이 적어 위협이 없는 국면)에서 `evaluate()`가 거의 0 근처로만 나와 탐색이 후보들을 사실상 무차별하게 고르는 문제를 지적받아, `positionalScore`(중앙에 가까운 돌일수록 가산점, 체비셰프 거리 기준 8~1점)를 `evaluate()`에 추가.

- P0 회귀·`vite build` 통과
- 자가대국 12판(300ms): **5승 7패** — 오히려 살짝 불리
- **Yixin 벤치마크(렌주룰+백, 2000ms) 6판: 0승 6패, 평균 20.0수(베이스라인 23.0수보다 오히려 더 빨리 짐)** — 개선 없이 손해, `git restore`로 되돌림(미커밋)

### Yixin 사고시간 핸디캡 실험 (2026-07-04)

다섯 번 연속 무효 결과의 공통점(항상 0승, 15~29수 안에 끝남)이 "벤치마크가 바닥에 붙어있어 민감도가 없는 것"일 수 있다고 판단, `tools/bench-vs-yixin.mjs`에 `YIXIN_TIME_BUDGET_MS`(Yixin 전용 사고시간, 커맨드라인 3번째 인자로도 오버라이드 가능)를 분리 추가해 Yixin을 의도적으로 약화시켜봄.

- 2000ms→500ms(4배 감소): 여전히 0승 6패, 평균 22.3수(베이스라인 23.0수와 거의 동일)
- 500ms→50ms(추가 10배 감소, 총 40배): 여전히 0승 6패, 평균 19.0수(오히려 더 빨리 짐)
- **결론**: Yixin의 우위는 탐색 시간과 거의 무관하다. 별도로 같은 국면에 50ms/3000ms를 줘서 실제 소요시간·수를 비교해보니 수가 서로 달라(고정 오프닝 북은 아님) 순수 북은 아니지만, 이건 Yixin의 강함이 "더 깊이 읽어서"가 아니라 평가함수/패턴 지식 자체의 질에서 온다는 뜻 — 지금까지의 다섯 번 시도가 전부 "탐색을 더 잘/깊게" 계열이었던 것과 정확히 어긋나는 근본적인 문제 제기

### 연구 기반 오프닝·평가함수 재시도 및 되돌림 (2026-07-04)

Yixin이 강한 이유를 웹 검색으로 실제 조사(추측 아님): (1) [Renju opening pattern - Wikipedia](https://en.wikipedia.org/wiki/Renju_opening_pattern) — RIF 오프닝 분류상 백의 2번째 수가 흑돌에 대각선으로 붙는 "간접 오프닝"이 직교로 붙는 "직접 오프닝"보다 백에게 전반적으로 더 균형적/유리, (2) [Tournament-winning gomoku AI - sortingsearching.com](https://sortingsearching.com/2020/05/18/gomoku.html) — 대회 우승급 엔진은 "이미 놓인 돌의 모양"이 아니라 "각 빈 칸마다 거기 두면 생기는 가장 강한 위협 2개를 지수적으로 합산"하는 평가함수를 씀.

- `getOpeningMove`를 대각선 4방향으로만 제한(기존 8방향 랜덤에서 축소)
- `boardScore`/`WINDOW_WEIGHTS`를 `potentialScore`(빈 칸마다 4방향 중 최고/2등 위협 심각도를 `1.5·3^a + 3^b`로 합산)로 전면 교체. 최초 밑수 1.8(출처 그대로)로는 정적 검증에서 "막힌사 하나 < 열린삼 하나"로 단조성이 깨지는 걸 발견(열린삼의 양끝 두 자리 합이 인접 등급 하나를 역전) — 밑수를 3으로 올려 단조성 확보 후 통과
- P0 회귀·성능(2000ms 예산 내 정상)·`vite build` 통과
- 자가대국 12판(300ms): **4승 8패** — 이번에도 오히려 불리
- **Yixin 벤치마크(렌주룰+백, 2000ms, Yixin 500ms) 6판: 0승 6패, 평균 19.7수** — 실제 학술·업계 자료를 근거로 했음에도 개선 없음, `git restore`로 되돌림(미커밋)

### 종합: 여섯 번의 연속 시도 전부 Yixin 상대 무효/손해 (2026-07-04)

VCT, 시간예산 2배, 패턴 평가함수, 탐색기반 방어, 위치평가, 연구기반 오프닝+평가함수 — 여섯 가지 전부 나름의 근거(자가대국 검증, 정적 검증, 실제 렌주 이론·논문 자료)가 있었지만 Yixin 벤치마크에서 전부 무효였거나 손해였다. Yixin 사고시간을 40배 줄여도(2000→50ms) 결과가 거의 안 바뀐다는 점까지 확인돼, **이 여섯 번의 시도가 공유하는 "탐색을 더 잘/깊게 하면 나아진다"는 전제 자체가 근본적으로 재검토가 필요한 상태**.

### 자동 가중치 튜닝 시도 — 방법론 자체의 한계 재확인 (2026-07-04)

`classifyLine`(열린/막힌 구분) 아키텍처의 모양별 점수 가중치 5개 조합을 자동 생성해 자가대국 게임틀릿으로 비교하는 튜닝 도구를 만들었다(스크래치패드, 커밋 안 함).

- 1차(12판, 오프닝 6개): **5개 후보 전부 베이스라인 승리**(7승5패~9승3패) — 고무적으로 보임
- 2차(20판, 오프닝 10개로 확대, 상위 2개만 재검증): 1위였던 두 후보가 **9승11패, 10승10패로 반전** — 표본을 늘리자 우위가 사라짐
- **결론**: 오늘 자가대국으로 봤던 다른 모든 신호와 같은 패턴(소표본 낙관 → 큰 표본에서 소멸)이 튜닝 도구 자체에서도 재현됨. 가중치를 여러 개 비교하려면 후보당 20판 이상이 필요한데 후보 수가 늘면 비용이 배로 커져, 이 방식만으로 신뢰성 있게 좋은 가중치를 찾기는 비효율적

## 증분 평가함수 최적화 (2026-07-04)

지금까지의 6가지 시도(및 그 이전 "시간예산 2배" 실험)는 전부 "탐색을 더 잘/깊게" 계열이었고, 전부 Yixin 상대로 무효였다. 그런데 "2배" 실험은 사실 깊이 가설을 약하게만 검증했다 — 분기가 있는 트리에서 노드 수를 2배로 늘려도 실제 늘어나는 깊이는 1ply에 훨씬 못 미친다(depth ∝ log(노드 수)). 이번엔 시간예산이 아니라 **노드당 비용 자체**를 낮추는 무손실 리팩터를 시도해, "속도 최적화가 진짜 깊이를 늘리는지" + "그게 Yixin 상대로 의미가 있는지"를 분리해서 봤다.

- **문제**: `evaluate()`가 leaf 노드마다 `boardScore()`로 보드 전체(4방향×15×15 윈도우)를 처음부터 재스캔. leaf가 탐색당 수만 번 방문되는데 매번 O(board) 재계산.
- **수정**: 보드의 모든 5칸 윈도우를 미리 한 번만 열거해두고(`CELL_WINDOWS`), 각 윈도우의 흑/백 돌 개수만 배열로 유지. 착수/취소마다 그 돌이 속한 윈도우(최대 20개)만 갱신하고 누적 점수(`score1`/`score2`)도 함께 델타로 갱신 → `evaluate()`는 O(1) 조회. **점수 공식(`WINDOW_WEIGHTS` 합산 규칙)은 100% 동일** — 속도만 바꾼 것이며 평가 로직 자체는 손대지 않음.
- **정확성 검증**: 무작위 대국 300판(착수 최대 60수 + 매 판 무작위 undo 10회)에서 기존 재계산 방식과 증분 방식의 평가값을 매 수마다 전수 비교 — 21,300회 검사 전부 완전 일치(불일치 0건).
- **속도 측정**: baseline 자가대국에서 뽑은 실전 중반 국면 12개(9/15/21수 시점)에 동일 2000ms 예산을 주고 비교 — **평균 1.30배 더 많은 노드 탐색**(41,524~69,855 → 48,323~101,462), 12개 중 1개는 실제로 탐색 깊이가 1ply 더 깊어져(5→6) 다른 수를 선택함.
- **자가대국 회귀 확인** (300ms/수, 오프닝 6개×2색=12판): **6승 6패** — 완전 중립. 로직이 동일하니 예상된 결과(속도만 다르고 짧은 예산에선 깊이 차이가 거의 안 드러남).
- **Yixin 벤치마크(렌주룰+백, 2000ms, Yixin 500ms, 12판)**: baseline **0승 12패**(평균 26.5수) vs incr **0승 12패**(평균 24.8수) — **거의 동일, 유의미한 차이 없음**. 예상대로 "2배 시간예산" 실험과 같은 결론: 이 정도의 탐색 속도/깊이 개선(+30% 노드, 대부분 depth 불변)은 Yixin과의 실력차를 메우지 못한다.
- **결론 및 처리**: 강도(Yixin 승률) 개선 효과는 없었지만, **정확성 검증을 통과한 무손실 순수 속도 개선**(같은 로직, 같은 수 선택, 더 적은 CPU 사용)이라 다운사이드가 없어 프로덕션에 반영함(`vite build` 통과, 열린사 방어 등 스모크 테스트 통과). 이걸로 "탐색을 더 잘/깊게" 계열 시도는 시간예산 2배 실험에 이어 두 번째로, 그리고 이번엔 훨씬 큰 속도 배율로도 재확인됨 — **다음 시도는 이 계열을 벗어나야 함** (아래 "다음 후보" 참고).

## 렌주 금수/장목 활용 적용 (2026-07-04)

오늘 시도한 6가지가 전부 "탐색을 더 잘/깊게" 계열이었던 것과 달리, `aiEngine.js`에 렌주 금수(33/44/장목) 로직이 전혀 없다는 점(검색 결과 0건)에 착안해 아예 다른 전략 축을 적용함. AI는 항상 백이라 자기 금수 걱정은 없지만, 흑을 상대로 두 가지를 놓치고 있었다.

- **문제 1**: 백의 사(四)가 완성 지점을 1곳만 가질 때, 그 자리가 흑에게 금수(33/44/장목)면 흑은 원천적으로 못 막는데도(`server/index.js`가 승패 판정보다 금수 판정을 먼저 함 — 흑이 금수를 두면 5목 완성 여부와 무관하게 즉시 패배) `searchVCF`는 그냥 "상대가 막는다"고 가정하고 계속 재귀 탐색함
- **문제 2**: 흑이 6목 이상(장목)을 만드는 자리를 `hasImmediateWin`/`findCriticalDefenseCells`가 "즉시 승리"·"위협"으로 오판정 — 실제로는 흑의 즉시 패배 조건인데 백이 불필요하게 막으려 들 수 있었음

`client/src/utils/forbidden.js`의 `checkForbidden`을 `aiEngine.js`에 import해서 세 곳(`searchVCF`, `hasImmediateWin`, `findCriticalDefenseCells`)에 반영. **격리 단위 테스트**(같은 모양에서 금수 함정 있음/없음 대조군)로 오탐 없이 정확히 동작하는 것 확인 — 함정 있으면 확정승리로 인식(`searchVCF`가 `[{row,col}]` 반환), 함정 없으면 정상적으로 `null`(강제승리 아님).

### 검증
- P0 회귀(열린사 방어, 자기 VCF 우선), 성능(2000ms 예산 내 정상), `vite build` 전부 통과
- Yixin 벤치마크(렌주룰+백, 2000ms, Yixin 500ms) 6판: 0승 6패, 평균 23.7수 — 베이스라인과 큰 차이 없음. **다만 이건 예상된 결과** — 이 기능은 "사의 유일한 완성지점이 하필 상대 금수"라는 특정 조건에서만 드물게 발동하는 니치 전술이라, 오늘 실패한 여섯 시도(모든 국면에 일반 적용)와 성격이 다름. 6판 안에 그 조건이 안 나타났을 가능성이 높고, 격리 테스트로 로직 자체는 이미 검증됨
- 존재하지 않던 능력을 추가하는 것이라 다운사이드 없음(P0/회귀 전부 정상) — Yixin 벤치마크 무변화와 무관하게 채택

## 오프닝북 추가 (2026-07-04)

초반 5~7수까지 오프닝북을 만들어달라는 요청으로 웹 리서치부터 시작함. 렌주 정석 오프닝(26개 명명 오프닝, RIF 분류)이 실제 존재하고 밸런스 등급도 있지만, 좌표 데이터가 전부 이미지 파일에만 있어 텍스트로 추출 불가했고, 결정적으로 그 밸런스 등급 자체가 **스왑 규칙이 있는 대회 프로토콜**을 전제로 매겨진 것이라 이 프로젝트의 단순 교대 규칙에 그대로 옮기기 어려움을 확인함([Renju opening pattern - Wikipedia](https://en.wikipedia.org/wiki/Renju_opening_pattern), [26 Renju Openings - RenjuNet](https://www.renju.net/openings/)).

대신 **로컬 Yixin에게 직접 질의**하는 방식으로 전환 — 이미 이 세션에서 진짜 강하다고 검증한 엔진의 실전 판단을 그대로 가져오는 게 불확실한 이론보다 신뢰도 높다고 판단.

- 스크래치패드에 `generate_book.mjs` 작성 — pbrain `BOARD` 명령으로 특정 국면을 Yixin에 주고 "이 색이면 뭘 두겠냐"를 질의(질의마다 새 프로세스로 상태 오염 방지). 백의 2수째 각각에 대해 Yixin이 흑/백을 번갈아 4수 더 진행하며 백의 4/6수째 응수를 기록
- `client/src/utils/openingBook.js` 신규: 각 국면을 `"row,col,player"` 정렬·결합 문자열로 키를 만들어 매칭. `getAIMove`가 즉시승리 체크 다음, VCF/방어보다 먼저 조회 — 정확히 일치하는 국면에서만 사용하고 안 맞으면 그냥 null이라 항상 정상 탐색으로 안전하게 폴백
- 단위 테스트: 북 라인 재현 시 1ms 만에 정확한 북 수 반환, 매칭 안 되는 국면은 정상적으로 2000ms 탐색 진행 확인
- P0 회귀·`vite build` 통과
- **Yixin 벤치마크(렌주룰+백) 6판**: 0승 6패, 평균 24.3수(베이스라인 23.0수와 비슷) — 다만 **실전 수순을 직접 확인하니 판1에서 북이 연속 두 번 정확히 발동**(흑(7,7)백(6,8)흑(5,8) 다음 정확히 북의 백(6,7), 이어서 흑(6,6) 다음 정확히 북의 백(5,5)) — 설계대로 동작하는 것을 실전 대국에서도 확인. 나머지 판은 흑(Yixin)이 다르게 둬서 미매칭(설계상 정상 — 안 맞으면 그냥 정상 탐색)
- 다운사이드 없는 추가라 채택
- **커버리지 착오 발견·수정**: 처음엔 "백의 2수째는 대각선 4방향뿐"이라고 착각해 대각선 4개 라인(8개 항목)만 만들었는데, 실제 `getOpeningMove`는 상하좌우 포함 8방향 전부 무작위로 쓴다는 걸 나중에 확인(대각선 제한은 이전 세션에 시도했다가 무효 판정으로 `git restore`된 변경이었음 — 코드엔 반영 안 돼 있었음). 즉 대각선 4개 라인은 백의 2수째 결과 중 50%만 커버하고 나머지 50%(직교)는 북이 있어도 절대 안 맞는 상태였음. 직교 4방향도 동일하게 Yixin 질의로 라인 생성해 **8방향 전부, 총 16개 항목**으로 확장
